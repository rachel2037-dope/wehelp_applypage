個人簡介

我畢業於日本慶應義塾大學法律系。返台後通過律師高考，並於今年 8 月完成 5 個月的實習。實習期間，在嚴格時限下負責判決與法規的檢索、法律意見撰寫、訴訟證據整理，養成「快速釐清問題、系統性學習新概念」的工作習慣。我希望把這套方法帶入往後學習網頁開發上。

申請動機

律師實習期間，我觀察到兩個高頻痛點：

  • 判決檢索仍仰賴關鍵字海撈與人工比對。市面上結合「法律 × AI」的新興工具常未切中爭點，且幻覺問題嚴重。
  • 在大量、重複的資料中人工篩選證據並摘要重點，不但耗時且易生疏漏。

在思考如何讓律師把時間轉移到更高價值的辯護策略時，我意識到若不結合科技，難以根本解決上述重複勞動的問題。學會寫程式，不僅可以做出prototype快速驗證我的想法，我也發現自己對「從零到一、以程式解決人類痛點」更有興趣，因而萌生投入能被多人使用、實際解決需求、提升個人工作效率相關產品開發工作的念頭。

WeHelp 的「任務導向、每週產出」與我備考律師時「每週參與讀書會、逼迫自己將重心放在輸出」的學習模式一致（詳負面情緒處理），也能藉由達標每週任務模擬軟體工程師的工作模式，檢驗自己是否適合，這些都十分吸引我。我也希望透過參與 WeHelp，結識正在轉職的夥伴，一起進步。

加入 WeHelp，我相信在嚴格明確的節奏與同儕回饋下，我更能培養自學或其他途徑較難達成的產品開發能力，實際產出作品集、成功轉職。

曾經做過哪些軟體工程技術相關的學習？若有作品請分享給我們。

在大學時期，我曾在一門 App 開發的 seminar 中學習基礎C++（約至 class 程度），期末專題則使用 Qt Creator 用現成元件開發了一個簡單的 Android App（示範影片請見此）。

此外，今年 9 月起，我每週五會到北教大旁聽「前端工程設計」課程，目前目前已參與四週，接觸到基礎的 HTML、CSS 以及 Figma 操作。

如果參與這個訓練，會怎麼安排學習時間？

在確實完成每週指定任務的基礎上，並行CS基礎打底。以彭彭老師課程中提到的電腦科學知識（演算法、資料結構等）為主軸，以Harvard CS50課程進行系統性打底，遇到瓶頸時向老師、助教求助。因可全職參與，預期額外投入每週5-10小時在打底上，每週產出學習筆記。

軟體技術日新月異，如何確定投入的領域是正確有回報的？

對我來說，「正確且有回報」的定義是：投入在有市場需求，且能連結到我個人理想的領域。

在市場需求方面，我計劃透過多種方式驗證：例如在 GitHub trending 中觀察 star 數成長趨勢及更新頻率，整理目前活躍的技術；在求職網站上分析 JD，掌握企業普遍要求的技能；並參加人機互動、UIUX 等工作坊，與 PM、設計師、工程師交流，理解產業痛點。等程式能力更成熟後，我也打算參加黑客松，第一線參與企業題目，快速發現還需要補足的能力。

在個人理想方面，我希望學到的技能能與過去經驗結合，對法律科技的發展有所貢獻。同時，我也對音樂創作與人際溝通領域感興趣，希望能做出相關 demo（如幫助音樂新手生成簡易和弦進行、提供效果器體驗），再透過使用者回饋檢驗價值，若方向不對再快速調整。

請描述一件產生明顯負面情緒的經歷，如何處理該情緒？

準備律師考試初期，我從零學習台灣法律，連續兩年落榜，讓我一度陷入很深的自我懷疑與焦慮。

我嘗試把情緒與能力分離，並加入上榜前輩的讀書會仔細回顧學習流程。我將他的做法和自己的比對後，發現問題有：一，爭點繁多、容易遺忘；二，補習內容過雜過細、難以抓重點。

因此我調整策略：建立有意識、小範圍的輸出──每次讀完一個章節就用自己的話改寫重點筆記，並固定每週練考古申論題。為了避免陷入完美主義（背熟才動筆），我定時交作業，並對照高分答案訂正，藉此驗證自己抓重點的能力是否正確。這個方法幫助我逐步克服負面情緒，最終在第三年成功通過律師考試。

這段經驗讓我形成三個習慣：以成功者的經驗調整努力方向（避開彎路）、輸出base的學習（讓進度可視化、也有成就感）、用自己的語言建構知識（降低遺忘機率）。在Bootcamp中，我會沿用相同策略，在遇到挫折時，用可行的小行動化解焦慮與自我懷疑。

關於這份申請網頁，分享一個開發時的技術心得。

這次實作讓我意識到，真正寫程式的時間其實不多，更多心力其實花在構思上：產生要放的內容、規劃版面與切版、思考哪些地方需要加上互動或視覺效果。前期如果沒有把架構與流程想明白，很容易寫出冗長、難維護、邏輯又不夠清楚的程式碼。同時我也發現，評估自己的現階段能耐（能把頁面做到多精緻），其實並不容易。

過程中我有借助 ChatGPT 產生範例碼，但發現偏複雜、不易理解。因此改成對照彭彭老師的 YouTube 教學，結合在北教大旁聽前端課學到的觀念，逐行拆解與重構。看著頁面從極簡逐步長出層次與結構，是這次最有成就感的部分。

而印象深刻的技術抉擇包括：

  • 標籤的選擇（什麼時候用 section 、什麼時候用 article ）。
  • 同一效果的多種寫法該如何取捨（margin-top / margin-bottom 相對於 margin-block 。
  • 樣式應該寫在容器還是元素上（寫在 main 與寫在 section 的視覺差異與後續維護）。

受限於時間，這次尚未完善版本管理流程；之後我想補齊版本管理的知識，並優化版面與程式結構。

如何看待人工智慧的發展對軟體工程師的影響？

我認為即使現在已有 vibe coding 的工具，如果缺乏底層程式邏輯，不僅難以給出精準的 prompt，也無法辨識程式錯誤、優化架構，甚至可能產生資安疑慮。我認為，人工智慧的價值在於幫助軟體工程師快速完成從 0 到 1 的過程，讓工程師能把更多心力放在批判性思維與程式優化，並更專注於如何轉譯使用者需求。以我在律所實習的經驗為例，我常使用 ChatGPT 或 Claude 初步搜尋法規資料，雖能大幅縮短搜尋時間，但最終仍需要由懂法律的人確認資訊的正確性、給出能進聚焦問題的prompt，才能真正解決問題。同理，我認為人工智慧無法取代軟體工程師的判斷。
